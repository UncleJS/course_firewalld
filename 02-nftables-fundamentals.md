## 1. Why You Need to Understand nftables

You could use firewalld entirely through `firewall-cmd` without ever touching
`nft`. Many administrators do. But there are critical situations where deeper
knowledge is essential:

**Troubleshooting:** When a packet is being dropped and you can't figure out why,
inspecting `nft list ruleset` tells you exactly which rule is matching — with
packet counters. `firewall-cmd --list-all` shows your *intent*; nftables shows
*reality*.

**Understanding priorities:** firewalld generates rules with specific chain
priorities and ordering. Rich rules, zone rules, and policy rules all interact.
Without understanding chain priority numbers, you can't reason about which rule
wins a conflict.

**Container integration:** When Podman registers NAT rules, those rules appear
in nftables. Knowing how to read them is essential for understanding what's
actually happening to container traffic.

**Performance tuning:** Flowtables, sets, and maps in nftables offer significant
performance improvements. You need to understand the primitives to use them.

**Direct rules:** Module 12 covers writing nftables rules alongside firewalld.
This module is the prerequisite.

## 2. nftables Core Concepts

nftables is built on four core abstractions:

```
Tables
  └── Chains
        └── Rules
              └── Expressions → Match → Action
```

Additionally:
- **Sets** — collections of IP addresses, ports, or other values for efficient matching
- **Maps** — key-value structures for routing decisions
- **Flowtables** — hardware/software acceleration for forwarding

These are defined in a **ruleset** — the complete set of all tables, chains,
rules, sets, and maps currently loaded in the kernel.

## 3. Tables

A **table** is a top-level container that groups related chains. Every table
has:

1. A **family** — which protocol layer it applies to
2. A **name** — arbitrary, chosen by the creator

### Table families

| Family | Applies to |
|--------|-----------|
| `ip` | IPv4 packets only |
| `ip6` | IPv6 packets only |
| `inet` | Both IPv4 and IPv6 (most common for modern rules) |
| `arp` | ARP packets |
| `bridge` | Bridged traffic (layer 2) |
| `netdev` | Very early in packet ingress (before routing) |

firewalld uses the `inet` family almost exclusively, which means a single set
of rules handles both IPv4 and IPv6.

### Creating and deleting tables

```bash
# Create a table
nft add table inet my_table

# List all tables
nft list tables

# Delete a table (and all its contents)
nft delete table inet my_table
```

## 4. Chains and Hooks

A **chain** is an ordered sequence of rules. A chain that is attached to a
kernel **hook** is a *base chain* — it receives packets directly from the kernel.
A chain with no hook is a *regular chain* — it only receives packets via `jump`
or `goto` from another chain.

### Base chain hooks

| Hook | When packets arrive |
|------|---------------------|
| `prerouting` | All incoming packets, before routing decision |
| `input` | Packets destined for the local host |
| `forward` | Packets being forwarded (routed) through this host |
| `output` | Packets generated by local processes |
| `postrouting` | All outgoing packets, after routing decision |

### Chain types

| Type | Valid hooks | Purpose |
|------|-------------|---------|
| `filter` | prerouting, input, forward, output, postrouting | Accept or drop packets |
| `nat` | prerouting, input, output, postrouting | Address/port translation |
| `route` | output | Re-route based on marks |

### Chain priority

When multiple chains are attached to the same hook, **priority** determines
execution order. Lower numbers run first. Priority numbers are integers in the
range -2147483648 to 2147483647.

Standard named priorities (and their numeric values):

| Name | Value | Typical use |
|------|-------|-------------|
| `raw` | -300 | Before conntrack |
| `mangle` | -150 | Packet modification |
| `dstnat` | -100 | DNAT (prerouting) |
| `filter` | 0 | Standard filtering |
| `security` | 50 | SELinux/secmark |
| `srcnat` | 100 | SNAT (postrouting) |

firewalld uses specific priorities to place its chains in the correct order
relative to other software (containers, virtualization, VPNs).

### Creating chains

```bash
# Base chain (attached to a hook)
nft add chain inet my_table my_input_chain \
  '{ type filter hook input priority 0; policy drop; }'

# Regular chain (no hook — called via jump)
nft add chain inet my_table my_helper_chain
```

The `policy` on a base chain (`accept` or `drop`) is the **default verdict**
for packets that reach the end of the chain without matching any rule.

## 5. Rules and Expressions

A **rule** consists of:
1. Zero or more **match expressions** — conditions that a packet must satisfy
2. A **verdict/action** — what to do if all conditions match

### Basic rule syntax

```
nft add rule [family] [table] [chain] [match...] [action]
```

### Match expressions

nftables has a rich set of match expressions:

```bash
# Match on IP addresses
ip saddr 192.168.1.0/24          # IPv4 source address
ip daddr 10.0.0.1                # IPv4 destination address
ip6 saddr fe80::/10              # IPv6 source address

# Match on transport layer
tcp dport 80                     # TCP destination port
tcp sport 1024-65535             # TCP source port range
udp dport 53                     # UDP destination port
tcp flags syn                    # TCP SYN flag

# Match on protocols
ip protocol tcp                  # IPv4 protocol field
meta l4proto tcp                 # Protocol (works for IPv4 and IPv6)

# Match on interfaces
meta iifname "eth0"              # Input interface
meta oifname "eth1"              # Output interface

# Connection tracking state
ct state established,related     # ESTABLISHED or RELATED
ct state new                     # New connections only
ct state invalid                 # Invalid packets

# Match against a set
ip saddr @my_blocklist           # Source IP is in the named set
tcp dport { 80, 443, 8080 }      # Port is in an anonymous set
```

### Verdicts (actions)

| Verdict | Meaning |
|---------|---------|
| `accept` | Allow the packet; stop evaluating this chain |
| `drop` | Silently discard the packet |
| `reject` | Discard and send an ICMP error back to sender |
| `return` | Stop evaluating current chain; return to caller |
| `jump chain_name` | Evaluate another chain; return here if no match |
| `goto chain_name` | Evaluate another chain; do NOT return |
| `log` | Log the packet (does not stop evaluation) |
| `counter` | Count packets and bytes (does not stop evaluation) |

### Rule examples

```bash
# Accept established/related connections
nft add rule inet my_table my_input_chain \
  ct state established,related accept

# Drop invalid packets
nft add rule inet my_table my_input_chain \
  ct state invalid drop

# Accept SSH from a specific subnet
nft add rule inet my_table my_input_chain \
  ip saddr 10.0.0.0/8 tcp dport 22 accept

# Log and drop everything else (at end of chain)
nft add rule inet my_table my_input_chain \
  log prefix "DROPPED: " drop

# Accept HTTP and HTTPS
nft add rule inet my_table my_input_chain \
  tcp dport { 80, 443 } accept

# Reject with a specific ICMP message
nft add rule inet my_table my_input_chain \
  tcp dport 23 reject with icmp type port-unreachable
```

### Rule handles

Each rule gets a unique **handle** number when inserted. Handles are used to
reference rules for deletion or insertion at a specific position:

```bash
# List rules with their handles
nft -a list chain inet my_table my_input_chain

# Output example:
# table inet my_table {
#   chain my_input_chain {
#     type filter hook input priority 0; policy drop;
#     ct state established,related accept # handle 4
#     ct state invalid drop               # handle 5
#     tcp dport { 80, 443 } accept        # handle 6
#   }
# }

# Delete a specific rule by handle
nft delete rule inet my_table my_input_chain handle 5

# Insert a rule before handle 6
nft insert rule inet my_table my_input_chain handle 6 \
  ip saddr 192.168.0.0/16 accept
```

## 6. Sets and Maps

### Sets

A **set** is a collection of values (IPs, ports, etc.) that can be matched
against in a single rule. This is far more efficient than writing one rule per
value.

```bash
# Named set (persistent, can be modified at runtime)
nft add set inet my_table my_blocklist \
  '{ type ipv4_addr; flags interval; }'

# Add entries to the set
nft add element inet my_table my_blocklist { 192.168.1.100, 10.0.0.0/8 }

# Use the set in a rule
nft add rule inet my_table my_input_chain \
  ip saddr @my_blocklist drop

# Anonymous (inline) set — cannot be modified after creation
nft add rule inet my_table my_input_chain \
  tcp dport { 80, 443, 8080 } accept
```

### Set flags and types

```bash
# timeout flag: entries auto-expire
nft add set inet my_table temp_blocklist \
  '{ type ipv4_addr; flags timeout; timeout 1h; }'

# interval flag: allows CIDR ranges
nft add set inet my_table cidr_blocklist \
  '{ type ipv4_addr; flags interval; }'

# counter flag: count matches per element
nft add set inet my_table counted_ips \
  '{ type ipv4_addr; flags counter; }'
```

### Maps

A **map** is a key-value structure. The key is matched against a packet
property, and the value determines the action or destination.

```bash
# DNAT map: route different ports to different internal hosts
nft add map inet my_table port_to_host \
  '{ type inet_service : ipv4_addr; }'

nft add element inet my_table port_to_host {
  80  : 192.168.1.10,
  443 : 192.168.1.10,
  22  : 192.168.1.20
}

# Use the map in a NAT rule
nft add rule inet my_table prerouting \
  dnat to tcp dport map @port_to_host
```

## 7. The nft CLI

### Listing the ruleset

```bash
# Show everything
nft list ruleset

# Show a specific table
nft list table inet firewalld

# Show a specific chain
nft list chain inet firewalld filter_INPUT

# Show with rule handles (for deletion)
nft -a list ruleset

# Show with packet/byte counters
nft list ruleset   # counters appear if rules have counter statement

# Show sets
nft list sets
nft list set inet firewalld my_set
```

### Modifying the ruleset

```bash
# Add table
nft add table inet test_table

# Add chain
nft add chain inet test_table input_chain \
  '{ type filter hook input priority 10; policy accept; }'

# Add rule
nft add rule inet test_table input_chain \
  tcp dport 9999 drop

# Flush (clear) a chain — removes all rules but keeps chain
nft flush chain inet test_table input_chain

# Flush a table — removes all chains and rules
nft flush table inet test_table

# Delete a table completely
nft delete table inet test_table
```

### Atomic ruleset loading from a file

This is the safe way to load complex rulesets — it's all-or-nothing:

```bash
# Write a ruleset file
cat > /tmp/my_rules.nft << 'EOF'
table inet my_filter {
  chain input {
    type filter hook input priority 0; policy drop;
    ct state established,related accept
    ct state invalid drop
    iifname lo accept
    tcp dport 22 accept
    log prefix "DROPPED: " drop
  }
}
EOF

# Load it atomically
nft -f /tmp/my_rules.nft

# List to verify
nft list table inet my_filter
```

### Saving and restoring

```bash
# Save the complete current ruleset to a file
nft list ruleset > /tmp/ruleset-backup.nft

# Restore from file
nft -f /tmp/ruleset-backup.nft
```

> **⚠️  IMPORTANT — Don't save firewalld's ruleset and restore it manually**
> If firewalld is running, its tables are present in `nft list ruleset`. Saving
> and restoring this dump manually will conflict with firewalld's own management.
> Use this approach only for standalone nftables rulesets (without firewalld).

## 8. Connection Tracking in nftables

nftables integrates with the kernel's `conntrack` module for stateful inspection.
Connection states in nftables:

```bash
# The most common pattern: allow established/related, drop invalid, filter new
nft add rule inet my_table input \
  ct state established,related accept
nft add rule inet my_table input \
  ct state invalid drop
# ... specific rules for NEW connections ...
```

### Viewing the conntrack table

```bash
# See all tracked connections
conntrack -L

# Or with nftables monitor
nft monitor
```

### Connection tracking and performance

On a high-traffic server, connection tracking has overhead. nftables supports
`NOTRACK` to bypass conntrack for specific traffic:

```bash
# Don't track DNS responses (high-volume, stateless)
nft add rule inet my_table prerouting \
  udp sport 53 notrack
```

## 9. How firewalld Uses nftables

This is where the two worlds connect. Let's trace exactly what firewalld creates
in nftables when it starts.

### The firewalld tables

firewalld creates several tables when it starts. On RHEL 10 with a fresh config,
`nft list ruleset` shows:

```
# Table for core firewall filtering (IPv4 + IPv6)
table inet firewalld { ... }

# Table for NAT rules (IPv4 only — NAT doesn't apply to IPv6 the same way)
table ip firewalld { ... }
table ip6 firewalld { ... }
```

### The chain structure inside `inet firewalld`

firewalld creates chains for each hook, plus per-zone chains:

```
table inet firewalld {
  # Main entry points (base chains)
  chain filter_INPUT        # hook: input
  chain filter_FORWARD      # hook: forward
  chain filter_OUTPUT       # hook: output
  chain mangle_PREROUTING   # hook: prerouting, type mangle

  # Per-zone chains (one for each active zone)
  chain filter_IN_public     # Rules for the public zone (input)
  chain filter_FWD_public    # Rules for public zone (forward)
  chain filter_IN_internal   # Rules for the internal zone
  # ... etc for each zone ...

  # Per-policy chains
  chain filter_FWD_policy_policy_name  # Rules for named policies

  # Helper chains
  chain filter_IN_ZONES      # Dispatches to correct zone based on interface/source
  chain filter_FWD_ZONES     # Same for forward
}
```

### Tracing a packet through firewalld's chains

Let's trace an incoming TCP SYN packet destined for port 80 on a host with the
`public` zone on `eth0`:

```
Packet arrives at eth0
│
▼
inet firewalld : filter_INPUT (priority -1)
│   ct state invalid → drop
│   ct state established,related → accept
│
▼
inet firewalld : filter_IN_ZONES
│   iifname "eth0" → jump filter_IN_public
│
▼
inet firewalld : filter_IN_public
│   Check each allowed service/port
│   tcp dport 80 → accept   ← if http service is enabled
│   (no match)   → reject   ← default for public zone
```

### Seeing the zone dispatch rules

```bash
# Inside node1:
nft list chain inet firewalld filter_IN_ZONES
```

Output shows rules like:
```
iifname "eth0" goto filter_IN_public
```

This is how firewalld maps interfaces to zones in nftables.

### Correlating firewall-cmd with nftables

Let's add a service and see exactly what nftables rule gets created:

```bash
# Before
nft list chain inet firewalld filter_IN_public | grep -c "accept"

# Add HTTP
firewall-cmd --zone=public --add-service=http

# After
nft list chain inet firewalld filter_IN_public | grep "80"
# You will see: tcp dport 80 accept
```

This correlation — command → nftables output — is the core skill this module
builds.

### firewalld chain priorities

firewalld places its chains at specific priorities to coexist with other
software (containers, libvirt, VPNs):

| Chain | Priority | Reason |
|-------|----------|--------|
| `mangle_PREROUTING` | -15 | After most mangle, before filter |
| `filter_INPUT` | -1 | Just before default filter priority 0 |
| `filter_FORWARD` | -1 | Same |
| `filter_OUTPUT` | 0 | Standard output filter |
| `nat` chains | -100 (pre) / 100 (post) | Standard NAT priorities |

This means rules from other software at priority 0 run *after* firewalld's
rules — so firewalld's policy is applied first.

## 10. Flowtables — Software Fastpath

A **flowtable** is an nftables feature that moves packet forwarding decisions
into a fast, kernel-space shortcut. Once a connection is established and
classified, subsequent packets for that connection bypass the full chain
processing and are handled by the flowtable directly.

### When flowtables help

Flowtables are most beneficial on:
- Routers/gateways with high forwarding throughput
- NAT gateways handling many concurrent connections
- Hosts with complex, long rulesets (more rules = more flowtable benefit)

### Creating a flowtable

```bash
# Add the flowtable
nft add flowtable inet my_flowtable \
  '{ hook ingress priority 0; devices = { eth0, eth1 }; }'

# In the forward chain, offload established connections
nft add rule inet my_table forward \
  ct state established,related flow add @my_flowtable
```

### firewalld and flowtables

As of firewalld 2.x on RHEL 10, flowtable support is available but not
enabled by default. Module 12 covers enabling and tuning flowtables alongside
firewalld.

## 11. Atomic Rule Updates

One of nftables' most important advantages over iptables is **atomic ruleset
replacement**. When firewalld reloads, it doesn't:
1. Flush all rules (leaving the system momentarily unprotected)
2. Add new rules

Instead, it uses nftables transactions:

```bash
# Atomic multi-operation transaction
nft -f - << 'EOF'
flush table inet my_filter
table inet my_filter {
  chain input {
    type filter hook input priority 0; policy drop;
    ct state established,related accept
    tcp dport 22 accept
  }
}
EOF
```

The entire transaction succeeds or fails atomically. There is no window where
the system is unprotected mid-update.

This is why `firewall-cmd --reload` is safe to use on production systems even
while connections are active — the ruleset swap is atomic.

## Lab 2 — Write Rules Then Compare with firewalld

**Topology:** Single-node (node1 only)

**Objective:** Write a minimal but complete firewall policy using raw nftables,
then implement the same policy using `firewall-cmd`, and diff the two outputs.
This exercise makes the abstraction layer completely transparent.

---

### Step 1 — Start fresh and clear the slate

```bash
# 🔧 LAB STEP (inside node1)
podman exec -it node1 bash

# Stop firewalld so we can play with raw nftables without interference
systemctl stop firewalld

# Verify the ruleset is now empty (or has only system rules)
nft list ruleset
```

---

### Step 2 — Write a complete nftables ruleset from scratch

Our goal: allow SSH (22) and HTTP (80), drop everything else inbound.

```bash
# 🔧 LAB STEP

# Create the table
nft add table inet manual_firewall

# Create the input chain
nft add chain inet manual_firewall input \
  '{ type filter hook input priority 0; policy drop; }'

# Allow loopback
nft add rule inet manual_firewall input \
  iifname lo accept

# Allow established/related (replies to our outgoing connections)
nft add rule inet manual_firewall input \
  ct state established,related accept

# Drop invalid
nft add rule inet manual_firewall input \
  ct state invalid drop

# Allow SSH
nft add rule inet manual_firewall input \
  tcp dport 22 accept

# Allow HTTP
nft add rule inet manual_firewall input \
  tcp dport 80 accept

# Log everything else before the default drop
nft add rule inet manual_firewall input \
  log prefix "MANUAL-FW-DROP: " group 0
```

---

### Step 3 — List and verify

```bash
# 🔧 LAB STEP
nft list table inet manual_firewall
```

Expected output:
```
table inet manual_firewall {
  chain input {
    type filter hook input priority 0; policy drop;
    iifname "lo" accept
    ct state established,related accept
    ct state invalid drop
    tcp dport 22 accept
    tcp dport 80 accept
    log prefix "MANUAL-FW-DROP: " group 0
  }
}
```

### Step 4 — Test connectivity (from another node)

### Step 6 — Implement the same policy with firewall-cmd

The public zone starts with SSH allowed. We just need to add HTTP:

```bash
# 🔧 LAB STEP

# What does the public zone currently allow?
firewall-cmd --list-all --zone=public
# services: cockpit dhcpv6-client ssh

# Add HTTP
firewall-cmd --zone=public --add-service=http

# Verify
firewall-cmd --list-all --zone=public
# services: cockpit dhcpv6-client http ssh
```

---

### Step 7 — Compare the nftables output

```bash
# 🔧 LAB STEP

# Look at what firewalld generated
nft list table inet firewalld

# Focus on the public zone input chain
nft list chain inet firewalld filter_IN_public
```

> **💡 CONCEPT CHECK**
> Compare what you wrote manually vs what firewalld generated:
> - firewalld has many more chains (for zones, policies, helpers)
> - The established/related and invalid rules appear in `filter_INPUT`, not
>   per-zone chains — because they're global optimisations
> - The per-zone chain `filter_IN_public` contains the service-specific rules
> - firewalld adds rules for the `cockpit` and `dhcpv6-client` services too
>
> The structure is more complex but the logic is the same as what you wrote.

### Step 8 — Add a raw nftables rule and see where it fits

```bash
# 🔧 LAB STEP

# Add a rule for a custom port using raw nftables
nft add rule inet firewalld filter_IN_public \
  tcp dport 9999 accept

# Now verify via firewall-cmd
firewall-cmd --list-all --zone=public
# Notice: 9999/tcp does NOT appear in the firewall-cmd output

# But it IS in nftables
nft list chain inet firewalld filter_IN_public | grep 9999
```

> **⚠️  IMPORTANT — Direct nftables edits to firewalld tables**
> You CAN add raw nftables rules to firewalld's tables, but:
> 1. They are NOT visible to `firewall-cmd --list-all`
> 2. They ARE wiped on `firewall-cmd --reload`
> This is why Module 12 covers the proper way to add permanent raw rules via
> the direct rules interface or separate tables.
